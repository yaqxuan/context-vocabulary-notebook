# Context Vocabulary Notebook V1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the first version of Context Vocabulary Notebook: a local-first web app for manually creating video-context word sense cards, reviewing them with FSRS, managing tags/media, and importing/exporting data.

**Architecture:** Single repository with a React + Vite frontend, Node.js + Express backend, shared TypeScript types, and SQLite persistence through better-sqlite3. Backend owns domain invariants, filesystem media storage, FSRS scheduling, import/export, and soft-delete behavior; frontend consumes local API and keeps UI flows aligned with `require.md`.

**Tech Stack:** React, Vite, TypeScript, Tailwind CSS, Node.js, Express, SQLite, better-sqlite3, ts-fsrs, Vitest, React Testing Library, Playwright, JSZip.

## V1 Scope Guard

Do implement:

- Manual word sense card creation.
- Existing word sense suggestions by target word.
- Multiple context examples per card.
- Primary context rules.
- Sort order increments and up/down reordering.
- Local media files only: mp4, mp3, jpg, png, webp.
- Tags, search, filters, favorites, mastered/reviewing status.
- FSRS review with `Again` and `Good` only.
- Daily review limit as soft goal.
- Home, card creation, card list, card detail, review, tag management, favorites, statistics, settings pages.
- Marked zip export/import and pure-card zip export/import.
- Soft deletion.

Do not implement unless user explicitly expands scope:

- Built-in dictionaries.
- Phonetics or pronunciation data.
- AI automatic card creation.
- Card rollback/history.
- Source table.
- Website video links.
- Sync.
- Local API/CLI as user-facing extensions beyond internal web backend.
- MCP, OCR, ASR, subtitle extraction, browser extension, mobile/desktop packaging, automatic backup, Anki export.

## Cross-Phase Acceptance Criteria

The project is V1-complete only when all criteria below pass:

- [ ] User can manually create a word sense card from video context.
- [ ] User can add multiple context examples to one word sense.
- [ ] Existing target-word senses are suggested during card creation.
- [ ] One and only one active primary context exists per card when contexts exist.
- [ ] Lists and review show `暂无语境` when card has no active contexts.
- [ ] Context ordering uses `sort_order` increments and up/down swaps.
- [ ] Tags can classify cards and act as source markers.
- [ ] Search covers target word, contextual meaning, sentence, tag, and note.
- [ ] Review unit is card, not individual context or media.
- [ ] Review front shows full sentence and does not hide target word.
- [ ] Review has only Again and Good ratings.
- [ ] New cards immediately enter FSRS with due date equal to creation time.
- [ ] Mastered cards stay out of review queue and preserve FSRS state.
- [ ] Daily review limit is a soft reminder only.
- [ ] Media supports only local mp4/mp3/jpg/png/webp files.
- [ ] Missing imported media displays `文件不可用`.
- [ ] Cards, contexts, media, and tags use soft delete.
- [ ] Review logs are preserved but stats exclude logs from soft-deleted cards.
- [ ] Home page is action-oriented.
- [ ] Statistics page is analysis-oriented.
- [ ] Marked export preserves user-specific review/progress/settings data.
- [ ] Pure-card export excludes user-specific review/progress/settings data.
- [ ] Pure-card import reinitializes review/FSRS state.
- [ ] Import conflict handling supports all four required modes.
- [ ] Settings page exposes only V1 settings.
- [ ] No future extension is implemented without explicit user approval.

## Recommended Execution Mode After Approval

Use subagent-driven execution by phase:

1. Dispatch one implementation subagent for a phase or small vertical slice.
2. Run requested verification for that phase.
3. Request code review after code changes.
4. Fix review findings.
5. Move to next dependent phase only after validation passes.

No implementation should begin until user confirms this plan or requests changes.
